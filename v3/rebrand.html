Secure Send is now Data Padlock!

Why the branding change?

There are two reasons behind this branding change from Secure Send to Data Padlock:

The primary reason for the change is a major transformation in features that the add-on offers. In the new version of the add-on, the focus has been shifted from only sharing messages to now encrypting files. The name Secure Send implied that the add-on was geared towards securely sending messages, instead of the many other usages that it has, such as encrypting content for cloud storage. The name Data Padlock targets the broader mission of locking your data, not just text-based messages.

The second comes from a marketing standpoint. The name Secure Send was similar to Firefox's file sharing service Firefox Send. Because of this, searches online for "Secure Send" frequently resulted in Firefox Send outranking the add-on. The name change will improve the add-on's visibility and also help to reduce the chance of brand confusion between the add-on and Firefox Send.

What happened to sharing links?

+

This new version of the add-on has completely changed how messages or files are shared with other people. While previous versions of the add-on produced sharable links that automatically open in the add-on, this method was removed in favour of files. Now, the add-on produces a file that can be shared with other people.

There are a couple of reasons this feature is no longer in the newest version of this add-on:

The first is a technical limitation due to restrictions on the maximum length of a URL. Firefox (and other browsers) does have a limit to how long a URL can be. While most users may never have encountered this, users that were attempting to share longer messages would encounter issues when trying to open or share large links.

The obvious fix for this could have been to allow users to just copy and paste the encrypted text, but an issue with that approach is that the text could be misformatted by messaging services if it contained non-regular characters. This would make it impossible to read the original message. Making sure that the outputted text always contains regular characters is possible, but would dramatically increase the size of the text.

Instead, encrypting a message now produces a file that can be saved and sent to the intended recipient. This approach also integrates better with the new file encryption abilities that the add-on has.

What else is new?

+

In addition to what's already been mentioned above, the add-on has essentially been rebuilt from the ground up. This includes a brand new implementation of AES encryption that use's the browser's built-in encryption libraries instead of a third-party library. Unfortunately, this means that messages encrypted with an older version of the add-on cannot be opened with a newer version, but a complete archive of previous versions will remain available on both GitHub and the Mozilla add-on website.
